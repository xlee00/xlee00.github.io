ReactiveX响应式编程

ReactiveX 是一个专注于异步编程与控制可观察数据（或者事件）流的API。它组合了观察者模式，迭代器模式和函数式编程的优秀思想。

  • RxJava
    • 一个使用Observable序列实现包含异步和基于事件的编程方式框架。扩展自 观察者模式 ，实现数据、事件序列，并提供了很多Operators可以随意转换Observable，而无需顾虑线程安全，并发同步等。
    • 官方教程
    • 给 Android 开发者的 RxJava 详解
    • ReactiveX/RxJava文档中文版
    • Awesome-RxJava教程资源
  • RxAndroid
    • RxJava的Android扩展
    • 官方教程
  • RxLifecycle
    • 基于RxJava的Activity、Fragment生命周期控制组件。结合RxJava请求使用可以根据Activity、Fragment生命周期自动完成事件序列。避免需要手动调用unsubscribe。
    • 官方教程
  • RxBinding
    • Jake大神的大作, 用来将View的事件转换成Rx模式

事件总线

  • EventBus
    • EventBus是一款针对Android优化的发布/订阅事件总线。主要功能是替代Intent,Handler,BroadCast在Fragment，Activity，Service，线程之间传递消息.优点是开销小，代码更优雅。以及将发送者和接收者解耦。类似的库还有otto等。
    • 官方教程

####依赖注入

依赖注入是控制反转（Inversion of Control，英文缩写为IoC）的一种，控制反转就是关于一个对象如何获取他所依赖的对象的引用，这个责任的反转。当模块A中使用到了模块B，就称之为A对B产生了依赖。当我们将模块B在外面创建好再传入模块A，而不是之间在A内部创建B，就叫做依赖注入，通过依赖注入能实现模块之间的解耦。

  • ButterKnife
    • 简介：Butt erKnife 是一个专注于Android系统的View注入框架。ButterKnife 框架中使用的是『 Java Annotation Processing 』，会在编译时就解析注解，没有使用 『反射』去解析注解 ，从而避免了解析注解带来的额外性能的消耗。 在 ButterKnife. bind()方法中用到反射， 但是作者对每一个类进行了缓存，因此，不会对执行效率产生多大影响。 原理： http://www.tuicool.com/articles/N7fuui2
  • Dagger2
    • Dagger2是square公司Dagger的分支，是由谷歌公司fork Dagger并修改之后得到的，现在人们多使用Dagger2.
    • 官方教程
    • Dagger2 使用详解

数据存储

  • GreenDAO
    • GreenRobot组织开发的一个轻量快速的Android ORM数据库实现。
    • 官方教程
    • GreenDaoUpgradeHelper是一个greenDao的数据库升级帮助类。使用它可以很容易解决数据库升级问题，只需一行代码。
  • DBFlow
    • DBFlow，综合了 ActiveAndroid, Schematic, Ollie,Sprinkles 等库的优点。同时不是基于反射，所以性能也是非常高，效率紧跟greenDAO其后。基于注解，使用apt技术，在编译过程中生成操作类，使用方式和ActiveAndroid高度相似，使用简单。

数据缓存

  • LazyCacheForAndroid
    • 这是一个android上的数据缓存框架,具有缓存和加载数据速度快,缓存数据类型全,能够实现任意缓存时间等优点;具有二级缓存功能,可以自己定实现三级缓存;可扩展性高。教程见github。
  • ASimpleCache
    • ASimpleCache 是一个为android制定的 轻量级的 开源缓存框架。轻量到只有一个java文件（由十几个类精简而来）。 它可以缓存的东西： 普通的字符串、JsonObject、JsonArray、Bitmap、Drawable、序列化的java对象，和 byte数据。

如何选择框架

  • 如果框架A依赖另外的jar比较多，谨慎使用，学习也是要成本的
  • 如果框架B没有详细的文档，谨慎使用，理由同上
  • 如果框架C对你目前的App影响较大，改动的地方多，那么谨慎使用。
  • 如果框架D耦合度高，不方便扩展，谨慎使用。
  • 尽量使用专注、精简的框架，这样便于垂直学习，干扰很少，也便于拆分、砍掉.
  • 聚合型框架一定要放弃(比如Afinal，xUtils)，why？越是大而全，越容易牵一发而动全身，而且在框架世界里没有1+1>2这一说。相反的可读性差，耦合高，难扩展
  • 只有不断的阅读、学习良好的代码，才能提升自己代码质量。
  • 不要过多的引用类库，不要引用方法数过多的类库，以防范出现方法数65k的限制。
