3. 执行Rebase

除了merge类型的提交，其他类型提交都要执行Rebase

执行Rebase的时候会出现三种情况：

  • 直接执行完成。
  • 执行失败，不用处理。
  • 弹出提示（提示有一个选项，两个选项和三个选项一共三种类型），直接下一步就可以。

4. 开始review，点击Add Reviewer。

  • Reviewer可以为空，CC也可以为空
  • Code-Review:如果是普通用户只能+1分，总分要超过2分以后才能进行执行通过。否则无法通过审核。
  • Verified（普通用户没有这个选项）：通过审核，前提是Code-Review的总分超过2分，然后才能进行Verified。

当Verified获得1分以上的时候，主页面就会出现Submit（普通用户没有这个选项）
此时点击Submit：代码就会进入到仓库中，所有用户可以用git pull获得此更新。

五、其他

1. 创建新分支

通知卢佩芬或者具体仓库的管理员在网页后台添加分支。
添加分支需要提供某次commit 的Hash 或者一个分支，新分支基于此Hash或者分支创建新的分支。

2. 超大型提交

有时候我们会新建一个sdk仓库，首次提交的时候，会有大量的文件上传。由于大量文件修改，我们无法通过原来的方式上传代码。

这时候，我们需要让卢佩芬或者仓库管理员给我们开通直接上传代码不经过review的权限，进行代码上传，直接上传到服务器仓库。

代码上传完以后需要回收该权限，预防误操作。

六、Gerrit 仓库的工作模式

使用Gerrit code review，我们为什么要这么做git push gerrit HEAD:refs/for/master而不是做git push origin master

而使用git push origin master 会出现错误! [remote rejected] master -> master (prohibited by Gerrit)

工作流程

接收到一个仓库创建代码更新补丁的推送请求时，Gerrit更新数据库，然后将操作结果告知客户端。反馈一个成功的结果让客户端相信Gerrit已经创建了补丁，但实际上Gerrit根本没有在仓库中创建这个补丁。

一个成功push的补丁(即，补丁已经被推到Gerrit，将其放入“挂起的更改”暂存区域。)，Gerrit 经过review后，将更改从“未决更改”保存到 代码仓库，并根据它在你推送时提供的branch合并到代码仓库相应的branch中。这样以后，用户就能从权威存储库中下载被review的代码了。
